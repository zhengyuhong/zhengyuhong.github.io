# 用 Codex 写笔记的工作流

这个仓库现在有一个很清晰的职责：Markdown 是源文件，HTML 是生成结果，GitHub Pages 负责发布。以后写笔记时，只需要把注意力放在 `notes/` 里的内容上。

一篇新笔记从一个带日期的文件开始。文件名给它一个稳定的位置，front matter 给它标题、日期、标签和摘要，正文则保留 Markdown 的轻便和可读性。这样的结构足够简单，也足够让脚本自动生成首页、笔记页和标签索引。

Codex 在这里更像一个协作者：它可以帮我整理一个想法，补齐笔记结构，检查 front matter 是否符合约定，然后运行测试和构建。真正重要的是，笔记的源头仍然是可读的文本文件，不依赖某个编辑器或数据库。

发布也被压缩成一个动作：提交并推送到 `main`。之后 GitHub Actions 会运行同一个生成脚本，把 `site/` 发布到 `zhengyuhong.cn`。如果构建失败，网站不会被错误内容覆盖。

这条流程的好处是朴素：写作时像写文件，维护时像维护代码，发布时像推送一个小项目。笔记会随着一次次提交慢慢长出来。
